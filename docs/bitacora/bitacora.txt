Registro #0001 — Manifiesto Fundacional del Proyecto

Fecha: 2026-05-16
Autor: STRATEGIC QUANT COMMITTEE + Fundador del Proyecto
Tipo: Manifiesto / Dirección Estratégica / Documento Fundacional

Nombre Temporal del Proyecto

Pendiente de definición.

El proyecto será referido temporalmente como:

“Autonomous Quant Crypto System” (AQCS)

Este nombre es provisional y podrá modificarse posteriormente.

Propósito del Proyecto

Diseñar, construir y evolucionar un sistema cuantitativo autónomo de trading para crypto spot que sea:

robusto,
estadísticamente válido,
operacionalmente sostenible,
escalable,
y resistente a cambios de régimen de mercado.

El objetivo NO es construir:

un bot “mágico”,
un sistema hype-driven basado en IA,
ni una arquitectura innecesariamente compleja.

El objetivo es construir:

un sistema sobreviviente y evolutivo con fundamentos cuantitativos sólidos.

Filosofía Central

El proyecto adopta los siguientes principios:

1. Robustez sobre sofisticación

Se priorizarán:

estrategias simples,
explicables,
auditables,
y resistentes al overfitting.

La complejidad solo será introducida si demuestra mejora real y verificable.

2. Riesgo antes que retorno

La prioridad principal del sistema será:

supervivencia,
control de drawdown,
estabilidad operacional,
preservación de capital.

La maximización de retorno NO tendrá prioridad sobre la supervivencia.

3. Evidencia antes que narrativa

Toda hipótesis deberá validarse mediante:

backtesting riguroso,
walk-forward analysis,
pruebas out-of-sample,
Monte Carlo,
stress testing.

No se aceptarán afirmaciones basadas únicamente en:

intuición,
marketing,
redes sociales,
o resultados aislados.
4. Modularidad extrema

El sistema deberá diseñarse como componentes desacoplados:

Data Layer
Feature Engine
Signal Engine
Portfolio Construction
Risk Engine
Execution Engine
Monitoring & Logging

Esto permitirá:

escalabilidad,
reemplazo de componentes,
debugging eficiente,
auditoría técnica.
5. La IA no será el núcleo inicial

El proyecto NO comenzará utilizando:

LLMs para tomar trades,
Reinforcement Learning,
Deep Learning predictivo,
agentes autónomos no supervisados.

La IA solo será considerada posteriormente si demuestra:

valor incremental real,
robustez estadística,
y superioridad frente a baselines simples.
Mercado Objetivo
Mercado Inicial

Crypto Spot.

Registro #0002 — Decisión Estratégica de Arquitectura

Fecha: 2026-05-16
Autor: STRATEGIC QUANT COMMITTEE + Fundador del Proyecto
Tipo: Decisión Arquitectónica / Dirección Estratégica

Decisión

El proyecto adopta oficialmente una arquitectura híbrida:

Quant Core + LLM Oversight
Definición
Quant Core

Responsable de:

Ingesta de datos.
Cálculo de features.
Generación de señales.
Construcción de portafolio.
Sizing.
Risk management.
Ejecución simulada o real.

Debe ser:

determinístico,
auditable,
testeable,
reproducible.
LLM Oversight

Responsable de:

Monitoreo.
Explicación de decisiones.
Auditoría de anomalías.
Revisión de riesgo.
Generación de hipótesis.
Documentación automática.
Apoyo en research.

No tendrá permiso inicial para ejecutar trades directamente.

Restricción Crítica

El LLM no será fuente primaria de alpha en la V1.

Toda decisión de trade debe poder explicarse por reglas cuantitativas verificables.

Razonamiento

Esta arquitectura reduce el riesgo de:

alucinaciones,
decisiones no reproducibles,
sobreajuste narrativo,
comportamiento emergente,
reward hacking,
falta de trazabilidad.
Estrategia Base

Se mantiene como estrategia inicial:

Momentum + Trend Following Multi-Asset

Mercado:

Crypto Spot

Timeframes:

Daily como base robusta.
15m/4H solo como laboratorio posterior o estrategia separada.
Rol Correcto del LLM

El LLM podrá actuar como:

Research Analyst.
Risk Reviewer.
Strategy Auditor.
Documentation Agent.
Monitoring Agent.
Incident Explainer.
Experiment Planner.
Rol Prohibido en V1

El LLM no podrá:

abrir/cerrar posiciones directamente,
modificar riesgo sin reglas,
inventar señales no validadas,
cambiar parámetros en vivo,
operar por intuición,
tomar decisiones no auditables.
Recommended Strategic Direction

El proyecto debe avanzar en este orden:

Bitácora formal.
Arquitectura técnica.
Quant Core mínimo.
Backtesting.
Paper trading.
LLM Oversight pasivo.
LLM Oversight con permisos limitados.
Producción con capital pequeño.
What Should Be Avoided

Evitar:

“agentes traders” autónomos desde el inicio.
RL prematuro.
optimización agresiva.
LLM como decisor final.
15m como núcleo principal sin evidencia.
complejidad antes de validación.
Highest ROI Improvements

Prioridad inmediata:

Diseñar arquitectura modular.
Definir eventos/logs que el LLM podrá observar.
Definir límites de decisión del LLM.
Crear esquema de bitácora.
Construir baseline cuantitativo.
Roadmap
Fase 1 — Foundation
Bitácora.
Arquitectura.
Repositorio.
Estándares de logging.
Fase 2 — Quant Core
Data ingestion.
Feature engine.
Signal engine.
Portfolio engine.
Risk engine.
Fase 3 — Research
Backtesting.
Walk-forward.
Stress tests.
Fase 4 — Paper Trading
Trading simulado.
Logs completos.
Comparación live vs backtest.
Fase 5 — LLM Oversight
Agente observador.
Agente auditor.
Agente documental.
Agente de riesgo.
Fase 6 — Controlled Autonomy
Permisos limitados.
No ejecución directa.
Solo recomendaciones o reducción de exposición bajo reglas.
Final Verdict

Esta es una decisión sana y profesional.

Quant Core + LLM Oversight

maximiza aprendizaje y reduce riesgo.

Es una arquitectura mucho más defendible que intentar construir un trader autónomo basado directamente en LLM.
