### Design decisions

1. **Missing intervals use exact pd.date_range comparison** — mirrors the
   approach in `aqcs.data.validator._check_gaps`. Builds the expected date
   range and counts timestamps absent from the actual dataset. This is exact
   and deterministic (no diff-based approximation).

2. **Errors vs. warnings**:
   - **Errors** (cause `passed=False`): missing columns, non-UTC timestamps,
     non-monotonic timestamps, empty dataset, unreadable file, bad timeframe.
   - **Warnings** (advisory, `passed=True`): missing intervals, duplicate
     timestamps, NaN values, stale freshness lag.

3. **Logger removed** — `get_logger` was initially included but removed when
   the debug log mixed with JSON in `CliRunner` stdout. The `DataQualityReport`
   is self-documenting; callers may log from it at whatever level suits their context.

4. **`now_utc` injection** — Freshness lag uses wall-clock time by default.
   Tests inject a fixed `now_utc` to guarantee fully deterministic output.

5. **Value columns only in NaN count** — `nan_count_by_column` covers
   `open, high, low, close, volume`. Metadata columns (`symbol`, `timeframe`,
   `exchange`, `timestamp`) are excluded.

6. **`freshness_lag_seconds` in CLI determinism test excluded** — The CLI
   determinism test compares all fields except `freshness_lag_seconds`, which
   uses wall-clock time (no injection possible through the CLI without
   additional flags).

### Unresolved risks

- `freshness_lag_seconds` is non-deterministic when `now_utc` is not injected
  (CLI always uses wall-clock time). Acceptable for production monitoring;
  tests use injected `now_utc`.
- `pd.date_range` for `"1D"` uses calendar days. Datasets with DST-shifted
  offsets (non-UTC) will give wrong missing interval counts — but the UTC
  check rejects such data first, so this is not reachable.

### Known limitations

- Supported timeframes: `1m`, `5m`, `1h`, `4h`, `1d`. If a new timeframe is
  needed, add it to `_TIMEFRAME_SECONDS` and `_TIMEFRAME_FREQ`.
- `gap_policy` (halt vs. warn) is not configurable — warnings are always
  advisory. The research validation runner (`aqcs.research`) has its own
  `gap_policy` for that use case.
- No scheduling, alerting, or daemon behavior. This is intentional per scope.
